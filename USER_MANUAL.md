# Outreach App 運用マニュアル

## 1. システムの目的

「発信したいもの」と「刺さる人」をつなぐ営業支援ツールです。

イベント・サービス・コミュニティなどの告知を、受け手一人ひとりの文脈（過去の参加歴、興味、属性）に合わせてパーソナライズし、適切なチャネルで届けます。

### 自動化される部分と人間が介入する部分

| フェーズ | 処理内容 | 実行主体 |
|---------|---------|---------|
| 商材要素分解 | 商材の強み・切り口をAIが分析 | **自動**（AI） |
| カスタマー調査 | 各ターゲットとの関連度をスコアリング | **自動**（AI） |
| アプローチ文生成 | 受け手の文脈に合わせたメッセージをA/B案で生成 | **自動**（AI） |
| 承認（レビュー） | 生成されたメッセージを確認・編集・送信先を選別 | **人間** |
| 送信 | 承認されたメッセージをキュー経由で配信 | **自動**（システム） |

**重要: 人間が承認（Approve）するまで、メッセージが送信されることはありません。**

---

## 2. 基本ワークフロー

### Step 1: キャンペーンの作成

キャンペーンとは「何を、誰に、どう届けるか」の単位です。

以下の商材情報を入力します:

| 項目 | 説明 | 例 |
|------|------|-----|
| 名前 | キャンペーンの識別名 | 「伊勢の祭り 2026夏」 |
| タイプ | event / service / community / content | `event` |
| 説明 | 商材の詳しい内容 | 「1300年の歴史を持つ伝統祭事。3日間の...」 |
| URL | 詳細ページ | `https://...` |
| 日時 | 開催日時（イベントの場合） | `2026-07-15` |
| 場所 | 開催場所 | 「三重県伊勢市」 |
| タグ | 関連キーワード | `["伝統文化", "祭り", "地域"]` |

### Step 2: ターゲットの選定・条件設定

セグメント条件を AND/OR で組み合わせてターゲットを絞り込みます。

**使用できるフィルタ条件:**

| 条件タイプ | 説明 | 値の例 |
|-----------|------|--------|
| platform | 所属プラットフォーム | twitter, connpass, line, did-event |
| keyword | プロフィール・投稿のキーワード | 「伝統文化」「Web3」 |
| location | 地域 | 「東京」「関西」 |
| behavior | 行動履歴 | event-attended, post-about, follows |
| vc | Verifiable Credential の保有 | イベント参加証明など |
| token | トークン残高 | 最低保有数で指定 |

**条件の組み合わせ例:**

```
operator: AND
filters:
  - platform: "connpass"
  - behavior: "event-attended"
  - keyword: "文化"
```

→ Connpass でイベント参加経験があり、「文化」に関連するユーザーに絞り込む

**注意: 条件なし（全員対象）でのブロードキャストは禁止されています。**

### Step 3: AIによる自動分析と文章生成

キャンペーンを開始すると、以下の3ステップが自動で実行されます。

#### 3-1. 商材要素分解（decompose）

AIが商材情報を分析し、以下を抽出します:
- **コアバリュー**: この商材が提供する本質的な価値
- **ペルソナ別の切り口**: ターゲットの属性ごとに刺さるポイント
- **キーワード群**: 関連する検索・興味キーワード
- **差別化ポイント**: 他との違い
- **想定される反論**: 受け手が感じうる懸念

#### 3-2. カスタマー調査（analyze）

各ターゲットについて:
- 商材との関連度を **relevanceScore**（0〜100）で算出
- マッチする切り口（matchedAngle）と理由を特定
- スコアが閾値（デフォルト30）未満の人は自動的にスキップ

#### 3-3. アプローチ文生成（generate）

スコアが閾値以上のターゲットに対して:
- 受け手の文脈を主語にしたメッセージを**複数案（A/Bテスト用）**生成
- 150字以内で簡潔に
- 「ぜひ」「今すぐ」などの押し売り表現は自動的に排除

**ここまでが自動処理です。生成が完了すると、パイプラインは自動的に一時停止（paused）します。**

### Step 4: 人間の承認（Approve）

**これが最も重要なステップです。**

パイプラインが「review」ステップで一時停止すると、承認待ち状態になります。

#### 確認すべきポイント

1. **ターゲットの妥当性**
   - スコアリング結果を確認し、本当にアプローチすべき相手か判断する
   - 不適切なターゲットは承認対象から外す

2. **メッセージの品質**
   - 各ターゲットに対して生成された複数案を比較する
   - 受け手の文脈に合っているか確認する
   - 必要に応じてメッセージを編集する

3. **送信対象の選別**
   - 承認する場合、送信対象のターゲットIDリストを指定する
   - 全員を承認する必要はない。一部だけを選んで承認できる

#### 承認の操作

承認（approve）を実行すると:
- 選択したターゲットのIDリストがシステムに渡される
- パイプラインが再開し、送信ステップに進む

キャンセル（cancel）を実行すると:
- パイプライン全体が中止される
- 何も送信されない

**承認なしに送信が実行されることは、設計上ありません。**

### Step 5: 送信と結果のトラッキング

承認後、メッセージはキュー（BullMQ）経由で順次送信されます。

#### 送信チャネル

| チャネル | 状態 |
|---------|------|
| Gmail | 連携済み |
| LINE | 連携済み |
| WhatsApp | 連携済み |
| Twitter DM | 未実装 |

#### トラッキングされる結果

送信後、以下の結果が記録されます:

| アウトカム | スコア | 説明 |
|-----------|--------|------|
| converted | 100 | コンバージョン達成（参加登録、メンバー登録など） |
| replied | 70 | 返信あり |
| opened | 30 | 開封のみ |
| no-response | 0 | 反応なし |

## 3. 自己改善ループ

このシステムはキャンペーンの結果を次回の文章生成に活かす「自己改善ループ」を備えています。

### 仕組み

```
送信結果の蓄積
    ↓
成功パターン・失敗パターンの抽出
    ↓
現在のプロンプトの問題点を特定
    ↓
改善版プロンプトを自動生成
    ↓
次回のキャンペーンに適用
```

### 具体的な流れ

1. **結果集計**: 送信済みメッセージの結果（converted / replied / opened / no-response）をスコア化して集計
2. **パターン分析**: AIが成功例と失敗例を比較し、共通パターンを抽出
3. **プロンプト改善**: 現在のプロンプトの問題点を特定し、インクリメンタルに改善
4. **版管理**: 改善版プロンプトは `PromptIteration` として版番号付きで保存される

### 改善ループの特徴

- 大幅な書き換えではなく、少しずつ改善する（インクリメンタル）
- 改善の根拠が必ず記録される（changeNote）
- messaging.md のルール（受け手主語、150字以内、禁止ワード排除）は常に維持される
- キャンペーンを重ねるほど、メッセージの精度が向上する

## 4. 注意事項

### 自動送信の禁止

- **人間が承認（Approve）しない限り、メッセージは一切送信されません**
- システムは「review」ステップで必ず一時停止するよう設計されています
- この制約はステートマシンレベルで強制されており、バイパスできません

### 個人情報の取り扱い

**保持してよい情報:**
- 氏名（フルネームは避け、名前のみ推奨）
- メールアドレス（送信後は暗号化して保持）
- プラットフォームのユーザーID / URL
- 公開属性（職種・地域・興味タグ）
- 公開行動履歴（イベント参加、公開投稿）

**保持してはいけない情報:**
- 電話番号（送信時のみ使用、DB保存禁止）
- 住所
- 非公開のSNS情報
- 収集元が不明確な情報

**データ保持期間:**
- 未送信の Contact データ: 30日で自動削除
- 送信済みの Message データ: 90日で自動削除
- コンバージョンデータ（集計値のみ）: 無期限

### メッセージ生成のルール

- 受け手の文脈を主語にする（「イベントを開催します」ではなく「〇〇さんにとって役立つはず」）
- 150字以内を基本とする
- 「今すぐ」「必ず」「絶対に」「特別価格」「限定」（根拠なし）は禁止ワード
- 押し売りせず、選択肢として提示する

### 条件なしのブロードキャスト禁止

ターゲット条件を設定せずに全員へ送信することは禁止されています。必ずセグメント条件を設定してください。

### ログ出力に関する注意

console.log に個人情報を出力することは禁止です。デバッグ時はIDのみ使用してください。

## 5. パイプライン状態の一覧

| ステータス | 意味 | 次のアクション |
|-----------|------|---------------|
| pending | 開始前 | 自動で実行開始される |
| running | 処理中 | 完了を待つ |
| paused | 承認待ち | 内容を確認し、approve または cancel を実行 |
| completed | 全ステップ完了 | 結果をトラッキングする |
| failed | エラー発生 | エラー内容を確認し、再実行を検討 |
| cancelled | キャンセル済み | 必要に応じて新しいキャンペーンを作成 |

## 6. よくある質問

**Q: 承認せずに放置したらどうなる？**
A: パイプラインは paused のまま待機し続けます。送信は一切行われません。

**Q: 一部のターゲットだけ承認できる？**
A: はい。approve 時に送信対象のターゲットIDリストを指定するため、一部だけを選択できます。

**Q: 生成されたメッセージを編集できる？**
A: 承認前にメッセージの内容を確認・編集してから承認できます。

**Q: 改善ループはいつ実行される？**
A: キャンペーンの送信結果が蓄積された後、手動で改善リクエストを実行します。自動的に改善が走ることはありません。

**Q: 個人情報が漏洩するリスクは？**
A: 個人情報を含むファイルはリポジトリにコミットしません。テストデータは匿名化された fixtures を使用します。DB 内のメールアドレスは送信後に暗号化されます。
